# 1. 프로젝트 개요

## 1-1. 프로젝트 정체성

이 프로젝트는 사내에서 사용하는 **자체 개발 ERP**로, 다음 기술스택으로 구성된다.

| 영역 | 기술 |
|---|---|
| Backend | Python Flask (Blueprint 기반), psycopg + psycopg_pool |
| Frontend | Jinja2(최초 골격 렌더) + Vanilla JS(이후 SPA 방식) |
| Grid | AG Grid Community 31.0.0 (CDN 로드, Enterprise 기능 미사용) |
| DB | PostgreSQL 2종 (아래 1-2 참고) |
| 인증 | 세션 기반(Flask session, 서버 서명 쿠키) |
| 배포 | Docker / docker-compose |
| 버전관리 | 미사용 (git 대신 폴더 백업 방식) |

## 1-2. 이중 DB 구조

이 ERP는 **2개의 서로 다른 PostgreSQL DB**를 사용한다.

1. **로컬DB(`myappdb`)** — 이 애플리케이션 전용. 로그인 사용자 정보, 관리자 설정 등을 읽기/쓰기 모두 가능하게 관리.
2. **상용ERP DB(Odoo 기반)** — 사내에서 별도로 운영 중인 상용 ERP 시스템의 PostgreSQL을 **조회 전용**으로 연결. 계약/매출/매입/납품 등 실제 업무 데이터가 이곳에 있으며, 이 애플리케이션은 여기서 데이터를 조회해 손익분석 등 화면을 제공한다. 코드 레벨(`ReadOnlyConnection`이 SELECT만 제공)과 DB 계정 레벨(SELECT 권한만 부여, 인프라 설정) 이중으로 쓰기를 차단한다.

## 1-3. 핵심 설계 철학 — "엔진은 고정, 화면은 설정으로 추가"

이 프로젝트의 프론트엔드는 아래 원칙 위에서 만들어졌다.

> **SplitLayout 엔진 · TabManager · registry(등록소)는 한 번 작성한 뒤 절대 수정하지 않는다.**
> **새 화면을 추가할 때는 `SCREEN_LAYOUTS`(레이아웃 설정)와 `registerRenderer()`(렌더러 등록) 두 곳만 건드린다.**

이 원칙 덕분에 신규 메뉴 추가가 "엔진을 이해하고 고치는 일"이 아니라 "설정 데이터를 추가하는 일"이 되도록 설계되어 있다(자세한 절차는 [17장](./17_신규메뉴개발절차.md) 참고).

백엔드도 동일한 철학으로 **레이어드 아키텍처**(routes → service → repository → db, [4장](./04_Backend_Architecture.md) 참고)로 마이그레이션되어, 새 도메인을 추가할 때 기존 코드를 고치지 않고 파일만 추가하는 방식을 지향한다.

## 1-4. 현재 구현 상태 — Reference 화면 2개 기준

프로젝트 전체 메뉴 중 **실제로 백엔드까지 연동되어 동작하는 화면은 2개**뿐이며, 이 문서 전반에서 이 두 화면을 "Reference Architecture"(기준 구조)로 삼는다.

| 화면 | menuKey | 연동 DB | 상태 |
|---|---|---|---|
| 지급기준손익 | `profit_payment_date` | 상용ERP(조회 전용) | Backend/Frontend 연동 완료, 실 데이터 조회 가능 |
| 관리자 설정 | `admin_settings` | 로컬DB(읽기/쓰기) | Backend/Frontend 연동 완료, 실 CRUD 동작 |

나머지 메뉴(`account_register`, `biz_reg_no`, `voucher_entry`, `tax_invoice_search`, `vendor_info` 등)는 **UI 레이아웃/그리드 골격만 갖춘 Sample 화면**으로, 실제 데이터 연동 없이 레이아웃 참고용으로만 존재한다.

## 1-5. 이 문서의 목적과 읽는 법

이 개발계획서는 신입 개발자가 **이 문서만 읽고도** 프로젝트 전체 구조·개발 흐름·화면 동작 방식·데이터 흐름을 이해하고, 신규 메뉴를 스스로 추가할 수 있도록 작성되었다. 목차 순서대로 읽으면 "전체 그림(2~6장) → 프레임워크 상세(7~15장) → 실전 절차(16~19장)" 순으로 이해가 쌓이도록 구성했다.

문서 전반에 걸쳐 **⚠️로 표시된 항목은 문서(계획)와 실제 코드가 다르거나, 아직 해결되지 않은 이슈**다. 실제 소스코드를 직접 읽고 검증한 내용만 반영했으며, 특히 아래 항목은 신규 개발자가 반드시 먼저 인지해야 한다.

- ✅ (2026-07-18 조치 완료) `admin_settings`의 role 변경 API에 권한 검증이 없어, 로그인만 하면 누구나 자기 자신을 관리자로 승격시킬 수 있는 상태였으나, `auth.py`에 `role_required` 공통 데코레이터를 추가해 해결됨 (자세히: [16-3](./16_개발표준.md), [12장](./12_Reference화면분석.md), [18-A장](./18-A_보완개발진행현황.md))

다음 항목은 초기에는 문서-코드 불일치(⚠️)로 기록되었으나, 이후 프로젝트 담당자 확인을 거쳐 **의도적 설계 결정**으로 최종 확정되었다 — 버그가 아니라 정상 사양이니 참고만 하면 된다.

- ℹ️ `profit_payment_date` 탭을 열어도 자동조회가 되지 않는다. 실운영에 있어 자동조회가 불필요하다고 판단되어, 사용자가 직접 "조회" 버튼을 클릭한 경우에만 조회되도록 개발방향을 변경해 최종 개발을 완료했다. 관련 자동조회 코드는 삭제하지 않고 주석 처리된 상태로 남겨두었으며, 향후 필요해지면 주석만 해제해 재활성화할 수 있다 (자세히: [12장](./12_Reference화면분석.md))
