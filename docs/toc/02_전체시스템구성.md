# 2. 전체 시스템 구성

## 2-1. 시스템 구성도

```
┌──────────────────────────────────────────────────────────────────┐
│                          Browser (Client)                        │
│                                                                    │
│  base.html (최초 1회 서버 렌더)                                    │
│   └─ SPA 레이어: sidebar-tree.js → tab-manager.js → split-layout.js│
│       → screen-layouts.js(설정) → content-renderers/*.js(렌더러)  │
│       → api-fetch.js(공통 통신) → {도메인}-api.js                  │
└───────────────────────────┬────────────────────────────────────────┘
                             │ HTTP (JSON API / 최초 페이지 렌더)
┌───────────────────────────▼────────────────────────────────────────┐
│                        Flask App (Docker Container)                │
│                                                                     │
│  app.py (Blueprint 등록, CSRF, 로깅 초기화)                          │
│   ├─ routes/auth_routes.py   (로그인/로그아웃)                       │
│   ├─ routes/erp_routes.py    (대시보드, 지급기준손익 API)             │
│   └─ routes/admin_routes.py  (관리자 설정 API)                       │
│        ▼                                                           │
│  service/*.py  (비즈니스 로직: 검증, 화이트리스트, 날짜계산)            │
│        ▼                                                           │
│  repository/*.py  (SQL 작성/실행)                                    │
│        ▼                                                           │
│  db/local_db.py ─┐         db/prod_db.py ─┐                        │
└───────────────────┼─────────────────────────┼──────────────────────┘
                     ▼                         ▼
        ┌────────────────────┐    ┌──────────────────────────┐
        │  로컬DB(myappdb)    │    │  상용ERP DB (Odoo, 조회전용) │
        │  읽기/쓰기 가능      │    │  ReadOnlyConnection 강제    │
        └────────────────────┘    └──────────────────────────┘
```

## 2-2. 배포 구성

- **Docker / docker-compose**로 컨테이너화되어 있으며, `logs/` 폴더는 Docker 볼륨으로 마운트되어 컨테이너 재기동과 무관하게 로그가 보존된다.
- 정적 자원(css/js)은 `ASSET_VERSION`(프로세스 시작 시각 또는 환경변수)을 쿼리스트링으로 붙여 캐시버스팅한다 — 코드 수정 후 배포 시 브라우저 캐시를 수동으로 지울 필요가 없다.
- Nginx 등 별도 리버스 프록시 없이 Flask 컨테이너에 포트가 직접 매핑되는 구조로 확인됨([login_flow.md](../../login_flow.md) 기준).

## 2-3. 인증/세션 구성

- Flask의 서버 서명 세션 쿠키(`SECRET_KEY` 기반)를 사용. 로그인 성공 시 `user_id`/`email`/`role` 3개 값만 세션에 저장(최소 정보 원칙).
- `@login_required` 데코레이터(`auth.py`)가 인증 여부만 검사하며, **`role` 기반 인가(권한) 검사는 공통 데코레이터로 존재하지 않음** — 이 점이 시스템 전체에서 가장 중요한 미비점이다([16장](./16_개발표준.md) 참고).
- CSRF는 `flask_wtf.CSRFProtect`가 앱 전역에 걸려 있고, 프론트는 `<meta name="csrf-token">` → `api-fetch.js`가 자동으로 `X-CSRFToken` 헤더에 실어 보낸다.

## 2-4. 프론트엔드 렌더링 아키텍처(요약)

전체 흐름은 **"서버는 골격만, 화면 전환은 클라이언트에서"** 방식이다.

1. 로그인 후 `/dashboard`에서 `base.html`(앱 셸)과 사이드바 트리(`_sidebar_tree.html`)만 서버가 렌더링.
2. 이후 사이드바 클릭 → 탭 생성 → 레이아웃 분할 → 그리드/폼 렌더까지는 **전부 클라이언트 JS가 처리**, 서버는 JSON API 응답만 담당.
3. 이 구조 덕분에 화면 전환 시 페이지 전체 리로드가 없다(SPA와 유사하지만 React 등 프레임워크 없이 Vanilla JS로 구현).

상세 파이프라인은 [5장 Frontend Architecture](./05_Frontend_Architecture.md), [8장 TabManager 동작](./08_TabManager동작.md)에서 다룬다.

## 2-5. 백엔드 레이어드 아키텍처(요약)

```
routes(프레젠테이션, Blueprint) → service(비즈니스 로직) → repository(SQL) → db(커넥션 풀)
```

기존에는 `app.py` 하나에 라우팅/인증/DB 접근이 섞여 있었으나, Phase 0~5(레이어드 마이그레이션)를 거쳐 위 4단 구조로 전환 완료되었다(Phase 6 회귀검증, Phase 7 문서화는 계속 진행 중인 상태로 문서상 기록됨). 상세는 [4장 Backend Architecture](./04_Backend_Architecture.md).

## 2-6. 로깅 구성

모든 계층(routes/service/repository)이 `utils/logger.py`의 `get_logger(__name__)`을 통해 **동일한 로그 파일(`logs/app.log`, 5MB 로테이션 5개 보관) + 콘솔**에 통합 기록한다. Flask 자체 에러 로그도 같은 곳으로 위임되어 있어, 운영 중 문제 발생 시 이 파일 하나만 확인하면 된다.
