# 3. 폴더 구조

> 근거: [analysis/STEP2_폴더구조분석.md](../analysis/STEP2_폴더구조분석.md) (실측 전체 트리)

## 3-1. 최상위 구조

```
local_erp/
├── app.py                — Flask 엔트리 (Blueprint 등록만, 로직 없음)
├── auth.py                — 비밀번호 검증 + login_required 데코레이터
├── menu_data.py            — 사이드바 트리 데이터(DUMMY_TREE_MENU)
├── db/                     — DB 접속 계층
├── repository/             — SQL 쿼리 계층
├── service/                — 비즈니스 로직 계층
├── routes/                 — 라우팅(Blueprint) 계층
├── utils/                  — 공통 유틸(로거)
├── templates/              — Jinja2 템플릿(최초 골격)
├── static/{css,js,vendor}/ — 프론트 정적 자원
├── logs/                   — 런타임 로그 (Docker 볼륨)
├── plans/                  — 신규 기능 구현 계획 문서
└── docs/                   — 이 개발계획서(메인+analysis+toc)
```

## 3-2. 폴더별 역할과 규칙

| 폴더 | 레이어 | 규칙 |
|---|---|---|
| `db/` | DB 접속 | 순수 커넥션 풀 관리만. SQL 쿼리 절대 작성 안 함. `settings.py`(환경변수 정리) / `local_db.py`(로컬DB pool) / `prod_db.py`(상용DB pool, 조회전용 래퍼) 3개 파일로 고정 |
| `repository/` | 쿼리 | SQL 작성 전담. 도메인별 파일 분리(`local_repository.py`, `approval_repository.py`, `profit_analysis_repository.py`). DB 커넥션은 `db/` 계층 함수를 통해서만 획득 |
| `service/` | 비즈니스 로직 | routes는 이 계층만 호출, repository를 직접 호출하지 않음. 입력 검증/화이트리스트/날짜계산 등 "쿼리 밖 로직" 담당 |
| `routes/` | 프레젠테이션 | Blueprint 단위. `@login_required` 적용. HTTP 요청/응답 변환만, 비즈니스 로직 없음 |
| `utils/` | 공통 유틸 | 현재는 `logger.py`만 존재. 향후 공통 함수 추가 시 이 폴더 |
| `templates/` | 서버 렌더 뷰 | 최초 페이지 골격(`base.html`, `login.html`)만. 이후 화면은 전부 SPA |
| `static/js/` | 프론트 로직 | 엔진(`split-layout.js`, `tab-manager.js`) + 설정(`screen-layouts.js`, `search-fields-config.js`) + 도메인 렌더러(`content-renderers/*.js`) 3계층 분리 |
| `static/vendor/` | (미사용) | AG Grid는 CDN 로드 방식이라 실제로는 빈 폴더 |
| `plans/` | 기획 문서 | 신규 화면 구현 계획(개발계획서의 근거자료 중 하나) |
| `docs/` | 이 문서 | 메인(`ERP_개발계획서.md`) + `analysis/`(근거) + `toc/`(최종 산출물, 이 폴더) |

## 3-3. static/js 상세 — 엔진 vs 설정 vs 렌더러 분리

```
static/js/
├── (엔진, 절대 수정하지 않음)
│   ├── tab-manager.js        — 탭 생성/전환/닫기, Lazy Loading
│   ├── split-layout.js       — 레이아웃 트리 재귀 렌더, 드래그 리사이즈
│   └── content-renderers/registry.js  — registerRenderer, renderAgGrid, renderKeyValueForm
│
├── (설정, 신규 화면 추가 시 여기만 수정)
│   ├── screen-layouts.js         — SCREEN_LAYOUTS: menuKey → 레이아웃 트리
│   └── search-fields-config.js   — SEARCH_FIELDS_CONFIG: menuKey → 검색조건 필드
│
├── (공통 헬퍼)
│   ├── api-fetch.js       — 공통 apiFetch(), CSRF 자동첨부
│   ├── sidebar-tree.js    — 트리 토글/선택
│   ├── search-panel.js    — 검색조건 패널 빌더
│   └── toolbar.js         — 상단 툴바 "조회" 등 버튼
│
├── (도메인별 API 계층, eager 로드)
│   ├── admin-api.js
│   └── profit-analysis-api.js
│
└── content-renderers/  (도메인별 렌더러)
    ├── accounting.js        — account_register, voucher_entry (샘플)
    ├── tax-invoice.js        — biz_reg_no, tax_invoice_search, vendor_info (샘플)
    ├── profit-analysis.js    — profit_payment_date (Reference ①)
    └── admin.js               — admin_settings (Reference ②, Lazy Loading 대상)
```

## 3-4. 레거시/미사용 파일 (삭제하지 않고 보관 중)

git을 사용하지 않는 환경 특성상, 리팩터링 이전 원본이나 초기 자리표시자 파일이 삭제되지 않고 그대로 남아 있다. 신규 개발자는 아래 파일들이 **더 이상 사용되지 않는다는 점**을 인지해야 혼동하지 않는다.

| 파일 | 상태 |
|---|---|
| `db_legacy_backup.py` | 미사용 백업 |
| `templates/dashboard.html` | 코드에서 참조 제거됨, 파일만 존재 |
| `templates/base - 복사본.html` | 수동 백업본 추정 |
| `static/js/content-renderers.js.bak` | 리팩터링 전 원본(6개 메뉴가 한 파일에 있던 시절) |
| `static/js/splitter.js`, `static/js/grid-init.js` | 빈 파일(초기 자리표시자, 실제 구현은 다른 파일명으로 이동됨) |
| `static/js/_unused_date-default-rule.js.bak` | 미사용 백업 |

## 3-5. 신규 도메인/화면 추가 시 건드리는 폴더 (미리보기)

상세 절차는 [17장](./17_신규메뉴개발절차.md)에서 다루지만, 폴더 구조 관점에서 요약하면:

```
menu_data.py                    (트리 노드)
static/js/screen-layouts.js     (레이아웃)
static/js/content-renderers/{도메인}.js  (렌더러, 신규 도메인이면 파일 신규)
repository/{도메인}_repository.py
service/{도메인}_service.py
routes/{도메인}_routes.py + app.py Blueprint 등록
```

`db/`, `utils/`, 엔진 3종(`tab-manager.js`, `split-layout.js`, `registry.js`)은 신규 화면 추가와 무관하게 **수정 불필요**하다.
