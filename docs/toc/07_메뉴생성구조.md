# 7. 메뉴 생성 구조

> 근거: [analysis/STEP1_MD문서분석.md](../analysis/STEP1_MD문서분석.md) 1-9절(new_menu_dev_guide.md), [STEP4_Frontend분석.md](../analysis/STEP4_Frontend분석.md)

## 7-1. 메뉴 데이터의 출처 — `menu_data.py`

사이드바 트리메뉴는 `menu_data.py`의 `DUMMY_TREE_MENU`(현재는 3단계 더미 데이터)를 `routes/erp_routes.py`의 `dashboard()`가 `render_template("base.html", tree_menu=DUMMY_TREE_MENU)`로 넘겨 서버에서 1회 렌더링한다.

## 7-2. 트리 렌더링 — Jinja2 재귀 매크로

`templates/partials/_sidebar_tree.html`의 `render_tree(nodes)` 매크로가 자기 자신을 재귀 호출한다. 깊이 제한은 코드상 없으며, 노드는 "폴더"(하위 노드를 가짐)와 "리프"(실제 메뉴, `menuKey` 보유)로 나뉜다.

## 7-3. 리프 클릭 → 탭 생성 연결

```
사이드바 리프 클릭 (sidebar-tree.js, 이벤트 위임)
  → active 클래스 갱신
  → TabManager.openTab(menuKey, title)
```

즉, **메뉴 트리의 "생성"은 서버 데이터(`menu_data.py`) 몫이고, 메뉴 "클릭 시 동작"은 `menuKey` 하나로 이후 전체 파이프라인(탭→레이아웃→렌더러)에 연결**되는 구조다. 새 메뉴를 추가할 때 가장 먼저 정해야 하는 것이 이 `menuKey`이며, 이후 모든 설정 파일(`screen-layouts.js`, `search-fields-config.js`, `registerRenderer()`, `registerReload()`)이 이 값을 키로 사용한다.

## 7-4. 설정 버튼(admin_settings) — 트리 외부의 특수 진입점

`admin_settings`는 트리 메뉴에 있는 것이 아니라, 사이드바 하단에 고정된 ⚙ 버튼을 통해 진입한다.

```javascript
// sidebar-tree.js 세 번째 DOMContentLoaded 리스너
settingsBtn.addEventListener("click", () => TabManager.openTab("admin_settings", "설정"));
```

트리 리프 클릭과 **동일한 `openTab()` 함수**를 호출하므로 이후 파이프라인은 완전히 동일하다 — menuKey만 하드코딩되어 있다는 점이 유일한 차이다.

## 7-5. 메뉴 추가 시 필요한 최소 요소

| 요소 | 파일 |
|---|---|
| 트리에 표시될 노드 | `menu_data.py` |
| 클릭 시 레이아웃 | `static/js/screen-layouts.js` (`SCREEN_LAYOUTS[menuKey]`) |
| 검색조건(선택) | `static/js/search-fields-config.js` (`SEARCH_FIELDS_CONFIG[menuKey]`, 없으면 기본 폴백) |
| 실제 화면 렌더러 | `content-renderers/{도메인}.js`의 `registerRenderer(contentId, ...)` |
| 조회 버튼 배선(선택) | 같은 파일의 `registerReload(menuKey, fn)` |

이 5가지가 **메뉴 생성의 전부**이며, 엔진(`tab-manager.js`, `split-layout.js`, `registry.js`)은 건드리지 않는다. 상세 절차는 [17장](./17_신규메뉴개발절차.md)에서 예시와 함께 다룬다.
