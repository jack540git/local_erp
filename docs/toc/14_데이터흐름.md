# 14. 데이터 흐름

> 근거: [analysis/STEP7_데이터흐름분석.md](../analysis/STEP7_데이터흐름분석.md)

## 14-1. 앱 초기화 흐름

```
app.py 로드
  → setup_logging(app)     파일(logs/app.log, 5MB×5개 로테이션)+콘솔 핸들러 등록
  → ASSET_VERSION 계산      프로세스 시작 시각 → 정적파일 캐시버스팅
  → SECRET_KEY 로드         환경변수 FLASK_SECRET_KEY, 없으면 프로세스별 무작위 키 생성+경고로그(✅ 2026-07-18 조치, 아래 참고)
  → CSRFProtect(app)        전역 CSRF 검증
  → Blueprint 등록: auth_bp, erp_bp, admin_bp(/admin)
```

✅ (2026-07-18 조치 완료) 과거에는 `SECRET_KEY` 기본값이 `"dev-only-change-me"`로 하드코딩되어 있어, `.env` 미설정 시 모든 배포가 동일한 예측 가능한 키를 공유하는 위험이 있었다. 이제는 `FLASK_SECRET_KEY`가 없으면 `secrets.token_hex(32)`로 프로세스마다 무작위 키를 생성하고 `logger.warning()`으로 즉시 경고한다(고정 공유 값이 사라짐). 상세: [18-A장, ISSUE-02](./18-A_보완개발진행현황.md#2-2-issue-02-secret_key-하드코딩--완료).

## 14-2. 로그인 → 세션 → 화면 접근 전체 흐름

```
GET "/" (login_page)
  → 세션에 user_id 있으면 /dashboard로 리다이렉트, 없으면 login.html

POST "/login" (login_process)
  → find_user_by_email(email)   로컬DB, password 해시 포함 조회(로그인 시에만 예외적으로 필요)
  → verify_password(hash, pw)   bcrypt/werkzeug 자동분기
  → 성공 시: session.clear() → session[user_id/email/role] = DB 조회 시점 값
  → redirect(/dashboard)

GET "/dashboard" (@login_required)
  → session["email"], session["role"]을 base.html에 Jinja2로 주입
  → 이후 모든 화면 진입은 TabManager→SplitLayout→content-renderers 파이프라인(서버 재요청 없음)

POST "/logout"
  → session.clear() → 로그인 페이지로
```

### 🔴 세션 role 값의 정합성(Staleness) 문제
`session["role"]`은 로그인 시점 DB 조회 값으로 고정된다. admin_settings에서 관리자가 어떤 사용자의 role을 변경해도, **그 사용자가 이미 로그인해 있다면 재로그인 전까지 세션의 role 값은 갱신되지 않는다.** 현재는 라우트들이 `session["role"]`을 인가 검증에 쓰지 않아 실질적 위험은 낮지만, 향후 role 검증을 추가할 때 반드시 고려해야 할 설계 포인트다.

## 14-3. 요청 단위 데이터 흐름 종합

```
[Browser] → GET/POST (세션 쿠키 포함)
   ▼
[app.py] Blueprint 라우팅 → @login_required
   ▼
[routes/*.py] → 화면 렌더 또는 JSON API
   ▼
[service/*.py]  입력값 검증(정규식/화이트리스트), 권한(role) 검증은 미구현
   ▼
[repository/*.py]  SQL 작성 전담
   ▼
[db/local_db.py 또는 db/prod_db.py]  커넥션 풀 대여
   ▼
PostgreSQL (로컬 myappdb: 읽기/쓰기 | 상용ERP: 조회 전용, ReadOnlyConnection)
   ▼
JSON 응답 → apiFetch 표준 에러처리 → content-renderers가 그리드/폼 갱신
```

## 14-4. 로깅 데이터 흐름

- 모든 계층이 `get_logger(__name__)`으로 모듈별 로거를 얻어 **같은 파일(`logs/app.log`)+콘솔에 통합 기록**. Flask 자체 에러 로그도 동일 파일로 위임.
- 포맷: `%(asctime)s [%(levelname)s] %(name)s: %(message)s` — 모듈명이 그대로 찍혀 어느 계층 로그인지 구분 가능.
- ⚠️ `profit_analysis_repository.py`의 `logger.info("지급일기준 손익조회", date_payment)`는 `%s` 없이 인자를 넘겨, 실제 handler가 붙은 이 환경에서 `TypeError`를 유발할 가능성이 있다(로그가 이 지점부터 유실될 수 있으나 요청 자체는 계속 진행됨).

## 14-5. 2-DB 구조를 데이터 흐름 관점에서 비교

| | 로컬DB(myappdb) | 상용ERP DB(Odoo) |
|---|---|---|
| 흐름상 위치 | 로그인 인증, 관리자 설정 | 손익분석 등 조회성 화면 |
| 권한 | 읽기/쓰기 모두 허용 | `ReadOnlyConnection`으로 코드 레벨 조회 강제 |
| 커넥션 생성 시점 | 앱 시작 시 즉시 | Lazy(최초 호출 시, `.env` 미설정 시 호출 시점에만 에러) |
| 데이터 최신성 | 즉시 반영(같은 트랜잭션 흐름) | 상용ERP 운영 데이터 실시간 조회(별도 배치/동기화 없음) |

## 14-6. 신규 도메인의 데이터 흐름 설계 시 참고사항

- 로컬DB 대상이면 4-5(패턴 A, CRUD형)를, 상용DB 조회 대상이면 4-5(패턴 B, 분석/조회형)를 그대로 따른다([4장](./04_Backend_Architecture.md)).
- 상용DB에 새 조회를 추가할 때는 `ReadOnlyConnection`이 제공하는 `fetch_all`/`fetch_one`만 사용 가능함을 기억한다(쓰기 시도는 애초에 메서드가 없어 코드 작성 단계에서 즉시 발견된다).
