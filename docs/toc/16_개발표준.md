# 16. 개발 표준

> 종합 근거: STEP 1~7 전체(analysis/*.md)

## 16-1. 네이밍 규칙

| 대상 | 규칙 | 예시 |
|---|---|---|
| menuKey | 소문자+언더스코어 | `profit_payment_date`, `admin_settings` |
| contentId | `{menuKey}__{역할}_grid|form` | `profit_payment_date__list_grid` |
| DB 커넥션 함수 | `get_{local|prod}_db_connection()` (Flask `session`과 구분하기 위해 "connection" 사용) | `get_local_db_connection()` |
| Blueprint 변수 | `{도메인}_bp`, 등록 이름은 짧은 문자열 | `admin_bp` → `"admin"` |

## 16-2. 레이어 분리 원칙 (반드시 지킬 것)

1. `routes/*.py`는 `service`만 호출한다 — `repository`를 직접 호출하지 않는다.
2. `repository/*.py`는 `db/*.py`의 커넥션 함수를 통해서만 DB에 접근한다 — 자체적으로 커넥션 풀을 만들지 않는다.
3. `db/*.py`는 SQL을 작성하지 않는다 — 순수 커넥션/설정 관리만 한다.
4. 프론트의 모든 서버 통신은 `apiFetch()`를 거친다 — 직접 `fetch()` 호출 금지.
5. 엔진(`tab-manager.js`, `split-layout.js`, `content-renderers/registry.js`)은 수정하지 않는다 — 설정(`screen-layouts.js`, `search-fields-config.js`)과 신규 렌더러 파일만 추가한다.

## 16-3. 🔴 반드시 알아야 할 미해결 이슈 (신규 개발 시 반복하지 말 것)

### (1) 권한(role) 검증 공통 데코레이터 부재 — ✅ 조치 완료(2026-07-18)
~~`auth.py`에 `login_required`만 있고 `role_required`가 없다. `admin_routes.py`의 4개 라우트 전부 로그인 여부만 확인하며, 관리자 화면임에도 role 검증이 없어 **일반 사용자가 자기 자신을 admin으로 승격시킬 수 있다.**~~ (이전 서술, 이제 조치됨)

`auth.py`에 `role_required(*allowed_roles)` 공통 데코레이터를 신규 추가하고, `admin_routes.py`의 4개 라우트 전부에 `@login_required` 바로 아래 `@role_required("admin")`를 적용해 해결했다. 이제 role="user" 계정이 관리자 API를 직접 호출하면 `403`이 반환된다. 상세 변경 내역과 검증 절차는 [18-A장, ISSUE-01](./18-A_보완개발진행현황.md#2-1-issue-01-admin_routespy-role-검증-부재--완료) 참고.

> **신규 관리자 전용 API를 만들 때는 이제 `role_required("admin")`을 그대로 재사용하면 된다.** (과거 문서는 "데코레이터가 없으니 직접 체크하라"고 안내했으나, 이제는 공통 데코레이터가 존재하므로 그것을 사용하는 것이 표준이다.)

### (2) 로깅 코드의 포맷 인자 버그
```python
logger.info("지급일기준 손익조회", date_payment)   # ❌ %s 없이 인자 전달
logger.info("[손익조회] SQL 실행 (date_payment=%s)", date_payment)   # ✅ 올바른 패턴
```
`logger.info(msg, *args)`를 쓸 때는 `msg`에 반드시 `%s` 등 포맷 자리가 있어야 한다. 없으면 `TypeError`가 발생할 수 있다.

### (3) SECRET_KEY 기본값 하드코딩
```python
app.config["SECRET_KEY"] = os.environ.get("FLASK_SECRET_KEY", "dev-only-change-me")
```
운영 배포 시 `.env`에 `FLASK_SECRET_KEY`가 반드시 설정되어야 한다([19장 체크리스트](./19_체크리스트.md)).

### (4) 세션 role의 정합성(Staleness)
로그인 시점에 고정된 `session["role"]`은 이후 DB에서 role이 바뀌어도 갱신되지 않는다. 향후 role 검증을 추가할 때 세션 값을 그대로 신뢰하면 "권한 회수 지연" 문제가 생길 수 있다.

### (5) profit_payment_date 자동조회 미적용 — 미해결 이슈 아님(참고)
이것은 버그가 아니라 **의도적 설계 결정**이다. 실운영상 탭을 열 때마다 무거운 상용ERP 분석 쿼리가 자동으로 돌지 않도록, 사용자가 "조회" 버튼을 직접 누를 때만 조회되도록 최종 확정되었다. 해당 호출 코드는 삭제하지 않고 주석으로 남겨두었으므로, 향후 자동조회가 다시 필요해지면 주석만 해제하면 된다. 자세히: [12장](./12_Reference화면분석.md).

## 16-4. 신규 코드 작성 시 준수 사항

1. **DB 커넥션은 반드시 `with` 블록으로 사용**한다(`local_repository.py`의 모든 함수가 이 패턴).
2. **민감 컬럼(password 등)은 SELECT에서 명시적으로 제외**한다(`find_all_users`, `find_user_by_id`의 실례).
3. **입력값 검증은 service 계층에서** 한다(정규식, 화이트리스트) — routes나 repository에 검증 로직을 두지 않는다.
4. **AG Grid는 반드시 `renderAgGrid()`를 통해서만 생성**한다 — 공통 스타일/로케일 누락 방지.
5. **렌더러는 `{ render: function(el) {...} }` 형태로 등록**한다 — 함수를 직접 넘기면 안 된다(과거 전역 회귀 버그 원인, [9장](./09_ContentRenderer구조.md) 참고).

## 16-5. 레거시 파일 처리 원칙

git을 사용하지 않는 환경 특성상 과거 파일이 삭제되지 않고 남아있다(`*.bak`, 빈 파일 등, [3-4장](./03_폴더구조.md) 참고). 신규 개발자는 **이 파일들을 참고하거나 import하지 말 것** — 현재 사용 중인 파일과 이름이 유사해 혼동하기 쉽다.
