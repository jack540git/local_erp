# 17. 신규 메뉴 개발 절차

> 근거: [analysis/STEP1_MD문서분석.md](../analysis/STEP1_MD문서분석.md) 1-9절(`new_menu_dev_guide.md` 요약)

## 17-1. 예시 시나리오

신규 메뉴 `profit_loss`(유형 A, 단일 그리드, 회계 도메인 가정)를 추가한다고 할 때, 아래 순서로 진행한다.

## 17-2. 단계별 절차

### ① 트리 노드 추가 — `menu_data.py`
```python
{"type": "leaf", "menuKey": "profit_loss", "title": "손익계산서"}
```

### ② 레이아웃 등록 — `static/js/screen-layouts.js`
```javascript
SCREEN_LAYOUTS["profit_loss"] = { type: "pane", contentId: "profit_loss__main_grid" };
```

### ③ 렌더러 작성 — `content-renderers/{도메인}.js`
신규 도메인이면 파일을 새로 만들고 `base.html`에 `<script>` 태그 추가(`registry.js`보다 뒤, 다른 렌더러들과 순서 무관):
```javascript
registerRenderer("profit_loss__main_grid", {
    render: function (el) {
        const gridApi = renderAgGrid(el, columnDefs, []);
        registerReload("profit_loss", function (paneEl) {
            // 조회 로직: 검색조건 읽기 → API 호출 → gridApi.setGridOption("rowData", rows)
        });
    },
});
```

### ④ 검색조건 등록(선택) — `static/js/search-fields-config.js`
```javascript
SEARCH_FIELDS_CONFIG["profit_loss"] = [{ type: "date", name: "date", default: "closingDate" }];
```
없으면 기본 폴백(검색어 1개)이 자동 적용된다.

### ⑤ Backend — repository
```python
# repository/profit_loss_repository.py
from db.local_db import get_local_db_connection  # 또는 db.prod_db (조회 전용 대상이면)

def find_profit_loss(params):
    with get_local_db_connection() as conn:
        with conn.cursor() as cur:
            cur.execute("SELECT ...", (...,))
            return cur.fetchall()
```

### ⑥ Backend — service
```python
# service/profit_loss_service.py
from repository.profit_loss_repository import find_profit_loss

def get_profit_loss(params):
    # 입력값 검증(정규식/화이트리스트 등)
    return find_profit_loss(params)
```

### ⑦ Backend — routes + app.py 등록
```python
# routes/profit_loss_routes.py
profit_loss_bp = Blueprint("profit_loss", __name__)

@profit_loss_bp.route("/api/profit-loss")
@login_required
def profit_loss_api():
    ...

# app.py
app.register_blueprint(profit_loss_bp)   # ⚠️ 이 등록을 빠뜨리는 실수가 가장 흔함
```

### ⑧ Frontend — API 모듈 + base.html 등록
```javascript
// profit-loss-api.js
window.ProfitLossAPI = {
    get: (params) => apiFetch("/api/profit-loss?..."),
};
```
`base.html`에 `<script src=".../profit-loss-api.js">` 추가(도메인 렌더러보다 먼저 로드).

## 17-3. 절대 건드리지 않는 파일 (엔진)

```
static/js/sidebar-tree.js
static/js/tab-manager.js
static/js/split-layout.js
static/js/search-panel.js
static/js/content-renderers/registry.js
```

이 5개 파일은 "설정 데이터를 읽어서 동작하는 엔진"이며, 신규 메뉴 추가는 위 8단계처럼 **설정과 신규 파일 추가만으로 끝나야 한다.**

## 17-4. 체크리스트 (요약)

- [ ] `menu_data.py` 트리 노드
- [ ] `screen-layouts.js` `SCREEN_LAYOUTS`
- [ ] `content-renderers/{도메인}.js` `registerRenderer()` (+ `base.html` script 태그)
- [ ] `search-fields-config.js` (선택)
- [ ] `repository/{도메인}_repository.py`
- [ ] `service/{도메인}_service.py`
- [ ] `routes/{도메인}_routes.py` + **`app.py` Blueprint 등록**
- [ ] `{도메인}-api.js` (+ `base.html` script 태그)

## 17-5. 자주 하는 실수

1. `app.py`에 Blueprint 등록을 빠뜨림 (④에서 라우트를 만들어도 404가 남).
2. 렌더러를 함수로 직접 등록(`registerRenderer(id, function(el){...})`) — 반드시 `{ render: function(el){...} }` 객체로 등록해야 함([9장](./09_ContentRenderer구조.md)).
3. `contentId` 네이밍 규칙을 어김(`{menuKey}__{역할}_grid|form`) — 동작은 하지만 콘솔 경고가 뜨고 일관성이 깨짐.
4. 관리자 전용 화면인데 role 검증을 빠뜨림 — [16-3(1)](./16_개발표준.md)의 기존 이슈를 반복하지 않도록 주의.
