## 2. 진행 이력

### 2-1. ISSUE-01. `admin_routes.py` role 검증 부재 — ✅ 완료

**상태**: 🟢 완료 · **작업일**: 2026-07-18 · **관련**: [18-1절 1번](./18_향후개발계획.md#18-1-우선순위-최상--보안)

#### 문제 (변경 전)

`routes/admin_routes.py`의 관리자 전용 라우트 4개(`/admin/users`, `/admin/users/list`, `/admin/users/<id>`, `/admin/users/<id>/role`)에 `@login_required`만 걸려 있고, 호출자가 실제로 `role == "admin"`인지 검증하는 로직이 전혀 없었다. `service/user_service.py`의 `ALLOWED_ROLES` 화이트리스트는 **role 값**이 `admin`/`user` 중 하나인지만 검사할 뿐 **호출자 본인의 권한**은 검사하지 않았다.

즉, `role="user"`로 로그인한 일반 사용자도 아래처럼 직접 API를 호출하면 스스로 관리자로 승격할 수 있었다(STEP 3·5·6에서 코드로 최종 확증한 내용, [12장](./12_Reference화면분석.md) 참고).

```
POST /admin/users/{자기 user_id}/role
Body: {"role": "admin"}
```

#### 조치 (변경 내용)

| 파일 | 작업 구분 | 변경 내용 |
|---|---|---|
| `auth.py` | **수정(함수 추가)** | `role_required(*allowed_roles)` 데코레이터 신규 추가. 세션의 `role`이 허용 목록에 없으면 `403 {"error": "권한이 없습니다."}` JSON 응답을 반환. 기존 `login_required`는 변경하지 않고 그대로 유지(하위 호환). `jsonify` import 추가. |
| `routes/admin_routes.py` | **수정** | `from auth import login_required` → `from auth import login_required, role_required`로 import 변경. 관리자 라우트 4개(`user_settings_page`, `user_list_api`, `user_detail_api`, `user_role_update_api`) 전부에 `@login_required` 바로 아래 `@role_required("admin")`을 추가. |

**데코레이터 적용 순서(중요)**: `@login_required`가 `@role_required("admin")`보다 위(바깥쪽)에 있어야 한다. Flask/Python 데코레이터는 위에서 아래로 감싸는 순서로 실행되므로, `login_required`가 먼저 실행되어 "로그인 여부"를 확인한 뒤 `role_required`가 "권한(role)"을 확인하는 순서가 되어야 의미가 맞다. 순서를 바꾸면(즉 `role_required`를 위에 두면) 로그인 전에 세션의 `role`을 조회하게 되어 항상 `None`으로 걸러지는 등 의도와 다르게 동작할 수 있다.

```python
# auth.py 신규 추가분 핵심 (전체 코드는 파일 참고)
def role_required(*allowed_roles):
    def decorator(view_func):
        @wraps(view_func)
        def wrapped(*args, **kwargs):
            user_role = session.get("role")
            if user_role not in allowed_roles:
                return jsonify({"error": "권한이 없습니다."}), 403
            return view_func(*args, **kwargs)
        return wrapped
    return decorator
```

```python
# routes/admin_routes.py 적용 예 (4개 라우트 전부 동일 패턴)
@admin_bp.route("/users/<int:user_id>/role", methods=["POST"])
@login_required
@role_required("admin")
def user_role_update_api(user_id):
    ...
```

#### 변경 후 동작

- `role="user"`인 계정이 관리자 API 4개 중 하나라도 호출하면 **`403 Forbidden`** + `{"error": "권한이 없습니다."}` JSON 응답을 받는다. 더 이상 자기 자신을 admin으로 승격시킬 수 없다.
- `role="admin"`인 계정은 기존과 동일하게 정상 동작한다(기능 변화 없음).
- 로그인 자체가 안 된 사용자는 기존과 동일하게 `login_required`에 의해 로그인 페이지로 리다이렉트된다(순서 변경 없음).

#### 검증 방법 (자동화 테스트가 없는 프로젝트이므로 수동 검증 절차 기록)

1. `role="user"` 계정으로 로그인한 세션 상태에서 아래를 각각 호출해 전부 `403`이 반환되는지 확인한다.
   - `GET /admin/users`
   - `GET /admin/users/list`
   - `GET /admin/users/<임의 id>`
   - `POST /admin/users/<자기 id>/role` (`{"role":"admin"}`)
2. `role="admin"` 계정으로 로그인해 위 4개 요청이 기존과 동일하게 정상 동작(200)하는지 확인한다.
3. 미로그인 상태로 위 4개를 호출해 로그인 페이지로 리다이렉트되는지 확인한다(회귀 확인).

> 이 프로젝트는 `service/test_approval_service.py`처럼 단독 실행 스크립트 형태의 수동 테스트만 존재하고 pytest 등 자동화 테스트 스위트가 없다([STEP 1](../analysis/STEP1_MD문서분석.md) 참고). 정식 테스트 프레임워크 도입은 ISSUE-11과 별개로 향후 검토 대상.

#### 남은 작업 (후속 이슈로 분리)

- **프론트엔드 403 처리 미흡**: 사이드바 하단 ⚙(설정) 버튼은 role과 무관하게 항상 노출되어 있다([STEP 4](../analysis/STEP4_Frontend분석.md) 참고). 이번 조치로 일반 사용자가 admin_settings 탭을 열면 `AdminAPI.listUsers()` 등이 403을 받게 되는데, 프론트(`content-renderers/admin.js`, `admin-api.js`)에는 403 응답을 사용자 친화적으로 안내하는 처리가 없어 콘솔 에러 또는 빈 화면으로만 보일 수 있다. **보안 결함은 해소되었으나 UX는 아직 다듬어지지 않은 상태** — 별도 후속 항목(가칭 `ISSUE-01-A`: 관리자 아닌 사용자에게는 ⚙ 버튼 자체를 숨기거나, 403 수신 시 안내 메시지 표시)으로 18-1/18-2절에 추가 예정.
- ISSUE-06(세션 role staleness)과 연관: 로그인 중 관리자가 다른 사용자의 role을 변경해도, 대상 사용자의 세션은 재로그인 전까지 이전 role을 유지한다. 이번 조치로 "타인이 나를 승격/강등"시키는 시나리오의 신뢰 기준점이 세션이 되었으므로, ISSUE-06 해결 시 이 부분도 함께 고려해야 한다.

### 2-2. ISSUE-02. `SECRET_KEY` 하드코딩 — ✅ 완료

**상태**: 🟢 완료 · **작업일**: 2026-07-18 · **관련**: [18-1절 2번](./18_향후개발계획.md#18-1-우선순위-최상--보안)

#### 문제 (변경 전)

`app.py` 앱 초기화 코드가 아래와 같은 형태였다.

```python
app.config["SECRET_KEY"] = os.environ.get("FLASK_SECRET_KEY", "dev-only-change-me")
```

`.env`에 `FLASK_SECRET_KEY`가 설정되어 있지 않으면 **모든 배포/환경이 동일한 고정값(`"dev-only-change-me"`)을 세션 서명키로 공유**하게 되는 문제였다. 이 값이 외부에 노출되면 공격자가 이 키로 서명된 세션 쿠키를 위조(forge)해 임의 사용자로 로그인한 것처럼 가장할 수 있으며, CSRF 토큰도 같은 키로 서명되므로 함께 위험해진다(STEP 7에서 발견).

#### 조치 (변경 내용)

| 파일 | 작업 구분 | 변경 내용 |
|---|---|---|
| `app.py` | **수정** | 고정 문자열 폴백 제거. `FLASK_SECRET_KEY` 미설정 시 `secrets.token_hex(32)`로 **프로세스마다 무작위 키를 생성**하고, `logger.warning()`으로 즉시 경고를 남기도록 변경. `import secrets` 추가. |

```python
# app.py 변경 후 핵심
_secret_key = os.environ.get("FLASK_SECRET_KEY")
if not _secret_key:
    _secret_key = secrets.token_hex(32)
    logger.warning(
        "FLASK_SECRET_KEY 환경변수가 설정되지 않아 임시 키를 생성했습니다. "
        "이 임시 키는 프로세스 재시작 시마다 바뀌므로 기존 세션이 모두 무효화됩니다. "
        "반드시 .env 파일에 FLASK_SECRET_KEY를 설정하세요."
    )
app.config["SECRET_KEY"] = _secret_key
```

#### 변경 후 동작

- `.env`에 `FLASK_SECRET_KEY`가 설정된 경우: 기존과 동일하게 해당 값이 그대로 사용된다(변화 없음).
- **미설정인 경우**: 더 이상 고정값이 쓰이지 않고, 앱이 키를 가지는 시점(프로세스 로드 시점)마다 무작위 32바이트 키가 생성되며, **이 경우 `logs/app.log`에 경고가 남아 운영자가 배포 직후 즉시 인지할 수 있다**.
- 부작용: 임시 키는 컨테이너/프로세스가 재시작될 때마다 바뀌므로, **미설정 상태로 운영하면 배포/재기동마다 기존 로그인 세션과 CSRF 토큰이 전부 무효화**된다(사용자가 강제 로그아웃된다). 이는 보안상 안전한 방향이지만, **운영 안정성 관점에서는 `.env`에 `FLASK_SECRET_KEY`를 반드시 설정해야 함**을 의미한다([19장 체크리스트](./19_체크리스트.md)에 배포 전 확인 항목으로 반영 필요).

#### 검증 방법

1. `.env`에서 `FLASK_SECRET_KEY`를 제거한 상태로 앱을 기동해 `logs/app.log`에 경고 로그가 찍히는지 확인.
2. 이 상태에서 로그인 후 프로세스를 재시작하면 기존 세션이 무효화되어 다시 로그인이 요구되는지 확인(기대된 동작).
3. `.env`에 `FLASK_SECRET_KEY`를 설정한 후 재기동해도 세션이 유지되는지(정상 배포 환경 재현) 확인.

#### 남은 작업

- **배포 환경의 `.env` 실제 설정 여부는 미확인** — 코드는 안전하게 방어되어 있으나, 운영배포 담당자가 실제 `.env` 파일에 값이 설정되어 있는지 별도로 확인해야 한다([19장 체크리스트](./19_체크리스트.md) 항목 추가 권장).
- 경고 로그를 모니터링하는 운영 경보(알림 등)는 아직 없음 — 현재는 `logs/app.log` 파일을 사람이 직접 확인해야만 발견 가능하다.

### 2-3. ISSUE-07. `logger.info()` 포맷 인자 오류 — ✅ 완료

**상태**: 🟢 완료 · **작업일**: 2026-07-18 · **관련**: [16-3(2)절](./16_개발표준.md#16-3-반드시-알아야-하는-미해결-이슈-신규-개발-시-반복하지-말-것)

#### 문제 (변경 전)

`repository/profit_analysis_repository.py`의 `find_profit_by_payment_date()` 함수 첨두가 아래와 같은 형태였다.

```python
def find_profit_by_payment_date(date_payment: str):
    logger.info("지급일기준 손익조회", date_payment)   # ❌ %s 없이 인자 전달
    """
    지급기준일(date_payment) 기준 계약별 손익 목록 조회.
    ...
    """
```

Python `logging`은 `logger.info(msg, *args)` 호출 시 내부적으로 `msg % args` 포맷팅을 시도한다. `"지급일기준 손익조회"`에는 `%s` 자리표시자가 없는데 `date_payment`를 인자로 넘겼으므로, 실제 로그 기록 시점에 `TypeError: not all arguments converted during string formatting`가 발생할 수 있는 코드였다(STEP 6/7에서 발견). 다행히 logging 모듈이 이 예외를 `handleError()`로 자체 흡수해 API 응답 자체가 실패하지는 않았지만, **이 로그 한 줄이 매번 유실**되고 있었다.

부수적 문제: 이 `logger.info(...)` 호출이 함수 정의 바로 다음 줄(docstring이 와야 할 자리)을 차지하고 있어, 그 아래 문자열이 실제로는 docstring으로 인식되지 않고 그냥 실행되지 않는 문자열 리터럴 statement가 되어버리는 부수 문제도 함께 있었다.

#### 변경 후 동작

- `find_profit_by_payment_date()` 호출 시 더 이상 `TypeError` 위험이 없는 정상 포맷 로그가 찍힌다(`지급일기준 손익조회 (date_payment=2026-07-18)` 형태).
