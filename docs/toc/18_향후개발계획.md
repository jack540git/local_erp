# 18. 향후 개발 계획

> 근거: [analysis/STEP1_MD문서분석.md](../analysis/STEP1_MD문서분석.md) 1-10절(`future_considerations.md`), STEP 3~7 종합

## 18-1. 🔴 우선순위 최상 — 보안

1. **[✅ 완료] role 검증 공통 데코레이터(`role_required`) 추가** — `auth.py`에 추가 후 `admin_routes.py`의 4개 라우트에 적용. 현재 자기 자신을 admin으로 승격 가능한 취약점을 막는 최우선 작업([12장](./12_Reference화면분석.md), [16장](./16_개발표준.md)). **상세 작업 내역은 [18-A장, ISSUE-01](./18-A_보완개발진행현황.md#2-1-issue-01-admin_routespy-role-검증-부재--완료) 참고.**
2. **`SECRET_KEY` 운영값 강제** — `.env`에 `FLASK_SECRET_KEY` 미설정 시 앱 기동을 막거나 경고를 로그에 남기는 방어 코드 추가 검토.
3. **세션 role staleness 대응 설계** — role 검증 데코레이터 추가 시, 세션 값을 그대로 신뢰할지 매 요청 DB 재조회할지 결정 필요([14장](./14_데이터흐름.md)).

## 18-2. 🟡 우선순위 상 — 기능 완성도

1. **서브탭 실 데이터 연동** — `profit_payment_date`의 "매입"/"관련문서" 서브탭이 하드코딩 더미 상태. 선택된 행과 연동되도록 구현.
2. **`renderKeyValueForm` 수정 가능 모드 지원** — 현재 `readOnly` 고정이라 실제 CRUD 폼이 필요한 화면(vendor_info 등)에서 admin.js처럼 매번 직접 DOM을 구성해야 함. 공통 컴포넌트에 "읽기전용/수정가능" 모드를 추가하면 재사용성이 높아짐.
3. **툴바 나머지 버튼 배선** — 추가/삭제/저장/인쇄/닫기가 현재 `console.log`만 하는 더미 상태.

> 참고: `profit_payment_date` 자동조회는 미해결 과제가 아니다. 실운영 관점에서 자동조회가 불필요하다고 판단하여, 사용자가 "조회" 버튼을 직접 클릭했을 때만 데이터가 조회되도록 개발방향을 확정하고 개발을 완료했다([12장](./12_Reference화면분석.md) 참고). 따라서 본 향후 개발계획에서는 제외한다. 다만 향후 업무상 자동조회가 다시 필요해지면, 삭제하지 않고 남겨둔 주석 처리 코드를 해제하기만 하면 재활성화할 수 있다.

## 18-3. 🟢 우선순위 중 — 코드 품질/안정성

1. **로깅 포맷 인자 버그 수정** — `profit_analysis_repository.py`의 `logger.info("지급일기준 손익조회", date_payment)` ([16장](./16_개발표준.md)).
2. **Renderer Lifecycle(`destroy()`) 도입 검토** — 탭 반복 개폐 시 AG Grid 인스턴스/리스너 메모리 누수 가능성이 문서상 미해결로 남아 있으며 실측 검증도 없음([9장](./09_ContentRenderer구조.md)).
3. **레거시 파일 정리 여부 결정** — `db_legacy_backup.py`, `content-renderers.js.bak`, `splitter.js`, `grid-init.js`, `templates/base - 복사본.html` 등. git이 없는 환경이라 "삭제 대신 별도 archive 폴더로 이동" 등 정책을 정하는 것을 권장.
4. **응답 필드 정리** — `profit_payment_date` API 응답의 `total_cost2`/`profit2`/`profit_rate2` 계열이 화면에 쓰이지 않는데 계속 계산되고 있음. 용도를 확인 후 제거하거나 문서화.

## 18-4. 🔵 우선순위 하 — 확장 아이디어(설계안만 존재, 미구현)

`plans/admin_settings/04_schema_proposal.md`에서 제안된 스키마 확장안:

- `users.name`, `users.is_active`, `users.created_at`, `users.last_login_at` 컬럼 추가
- `user_login_history` 테이블(로그인 이력)
- `menu_permissions` 테이블(메뉴별 세부 권한 매트릭스, RBAC) — 현재는 role 2종(admin/user)만으로 동작하며, 화면 단위 권한 통제는 없음

## 18-5. 백엔드 도메인 확장 로드맵

`main_py_migration_structure_review.md`에 따르면, 상용 Odoo `main.py`(3,565줄) 기준 이관 대상 업무 도메인이 최소 9개(매출/매입/계약관리/손익조회/프로젝트수익/납품관리 등) 존재하며, `approval`(기안) 도메인처럼 **실제 필요한 화면부터 우선순위대로 하나씩** 이관하는 방식을 유지한다. 전체 일괄 이관은 계획되어 있지 않다.

## 18-6. 운영 참고사항

- 상용ERP 조회 부하 주의 — 일부 쿼리가 365일 기본 조회 범위를 가정하므로, 트래픽이 늘면 `statement_timeout` 외 추가 최적화(인덱스, 캐싱) 검토 필요.
- 컨테이너 타임존(Asia/Seoul) 반영 여부는 배포 환경마다 재확인 필요.
- `ASSET_VERSION` 캐시버스팅 쿼리스트링이 누락되지 않도록 신규 정적 파일 추가 시 `base.html` 템플릿 헬퍼 사용을 확인할 것.
