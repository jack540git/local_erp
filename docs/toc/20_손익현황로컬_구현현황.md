# 20. 손익현황(로컬) 구현 현황 (local_profit_data)

> 이 문서 하나만 읽으면 `local_profit_data`(손익현황-로컬) 화면이 **지금 어떻게 동작하는지** 전부 파악 가능하도록 작성함.
> 다른 신규 메뉴를 만들 때도 이 문서를 그대로 템플릿 삼아 참고할 수 있음 (`17_신규메뉴개발절차.md`의 실제 적용 사례).

## 1. 이 화면이 하는 일

사이드 트리메뉴 `손익관리 > 손익현황(로컬)`을 클릭하면 탭이 열리고, 검색조건(기준일자)을 입력한 뒤 "조회" 버튼(또는 Enter)을 누르면 **로컬DB(myappdb)** 를 조회해서 그리드에 표시한다.

동일 폴더의 `지급기준손익`(`profit_payment_date`)과 **화면 흐름은 완전히 동일**하고, 유일한 차이는 조회 대상이 상용ERP DB가 아니라 **로컬DB**라는 점뿐이다.

## 2. 관련 파일 전체 목록

| 계층 | 파일 | 역할 |
|---|---|---|
| 메뉴 등록 | `menu_data.py` | 트리메뉴에 `{"label": "손익현황(로컬)", "key": "local_profit_data"}` 등록 |
| 레이아웃 | `static/js/screen-layouts.js` | `SCREEN_LAYOUTS["local_profit_data"]` — 유형 A(단일 그리드) |
| 검색조건 | `static/js/search-fields-config.js` | `SEARCH_FIELDS_CONFIG["local_profit_data"]` — `date` 타입 필드 1개, 기본값 `closingDate`(마감일 자동계산 규칙) |
| API 모듈(프론트) | `static/js/local-profit-data-api.js` | `window.LocalProfitDataAPI.getByDate(date)` |
| 렌더러(프론트) | `static/js/content-renderers/local-profit-data.js` | 그리드 생성 + 조회 트리거 + destroy 정리 |
| Lazy Loading 등록 | `static/js/tab-manager.js` (`DOMAIN_SCRIPT_MAP`) | 위 렌더러 파일을 탭 최초 오픈 시에만 동적 로드 |
| 정적 리소스 로드 | `templates/base.html` | `local-profit-data-api.js`만 eager 로드 (렌더러는 Lazy라 여기 없음) |
| API 라우트 | `routes/erp_routes.py` | `GET /api/local-profit-data?date=YYYY-MM-DD` |
| 서비스 | `service/local_profit_data_service.py` | 날짜 형식 검증 + repository 위임 |
| 저장소 | `repository/local_profit_data_repository.py` | 로컬DB SQL 실행 |
| DB 접속 | `db/local_db.py` | (기존 파일, 수정 없음) `get_local_db_connection()` 재사용 |

## 3. 동작 흐름 (요청 1건 기준)

```
① 사용자: 사이드바에서 "손익현황(로컬)" 클릭
② sidebar-tree.js → TabManager.openTab("local_profit_data", "손익현황(로컬)")
③ tab-manager.js: 신규 탭 → ensureContentPane()
     - SearchPanel.build("local_profit_data")   → 기준일자 필드 생성, closingDate 규칙으로 기본값 자동 채움
     - isDomainRendererLoaded() → false(최초) → loadDomainScriptOnce()로 local-profit-data.js 동적 로드
     - 로드 완료 후 SplitLayout.render() → registerRenderer("local_profit_data__list_grid").render() 실행
         → renderAgGrid()로 빈 그리드 생성, listGridApi 저장
         → (자동조회는 하지 않음 — 사용자가 직접 "조회"를 눌러야 함, profit_payment_date와 동일 정책)

④ 사용자: 기준일자 확인 후 "조회" 버튼 클릭 (또는 필드에서 Enter)
⑤ toolbar.js: 활성 탭의 menuKey("local_profit_data") 확인 → ScreenReloadHandlers["local_profit_data"] 호출
⑥ local-profit-data.js의 loadLocalProfitData(paneEl) 실행
     - 검색조건 입력값 재검증(isValidYmd/formatYmd, search-panel.js 재사용)
     - window.LocalProfitDataAPI.getByDate(dateStr) 호출
⑦ local-profit-data-api.js: apiFetch("/api/local-profit-data?date=...") → GET 요청
⑧ routes/erp_routes.py: local_profit_data_api()
     - service.local_profit_data_service.get_local_profit_data_list(date) 호출
⑨ service: 날짜 형식(YYYY-MM-DD) 검증 → repository.local_profit_data_repository.find_local_profit_data(date) 호출
⑩ repository: get_local_db_connection()으로 로컬DB(myappdb)의 local_profit_data 테이블 SELECT
⑪ 결과가 JSON으로 응답 → ⑥의 .then(rows)에서 listGridApi.setGridOption("rowData", rows)로 그리드 갱신
⑫ 콘솔에 응답 데이터 로그, 서버(logs/app.log)에도 조회 로그 기록
⑬ (Enter로 트리거된 경우만) 입력 필드 blur 처리 — 조회 후 커서가 남아있는 어색함 제거
```

## 4. 사전 준비 — DB 테이블 (⚠ 아직 미생성)

이 프로젝트는 마이그레이션 도구(Alembic 등)가 없어서, 아래 DDL을 **로컬DB(myappdb)에 최초 1회 수동 실행**해야 실제로 동작한다. 테이블 없이 조회하면 500 에러가 난다.

```sql
CREATE TABLE local_profit_data (
    id                SERIAL PRIMARY KEY,
    doc_id            VARCHAR(50),
    doc_name          VARCHAR(200),
    biz_done          BOOLEAN DEFAULT FALSE,
    expected_sale     NUMERIC(15, 0) DEFAULT 0,
    occurred_sale     NUMERIC(15, 0) DEFAULT 0,
    uncollected_sale  NUMERIC(15, 0) DEFAULT 0,
    profit            NUMERIC(15, 0) DEFAULT 0,
    profit_rate       NUMERIC(5, 1),
    total_cost        NUMERIC(15, 0) DEFAULT 0,
    biz_note          TEXT,
    criteria_date     DATE NOT NULL
);
CREATE INDEX idx_local_profit_data_criteria_date ON local_profit_data (criteria_date);
```

실행 방법:
```bash
docker compose exec local-erp psql -h host.docker.internal -U myapp_user -d myappdb
```

## 6. 이름 충돌 방지 — 다른 화면과 겹치지 않는 이유

| 항목 | 값 | 비고 |
|---|---|---|
| 전역 API 객체 | `window.LocalProfitDataAPI` | `ProfitAnalysisAPI`/`AdminAPI`와 별개 |
| Flask 라우트 함수명 | `local_profit_data_api` | `profit_analysis_payment_date_api`와 별개 |
| API URL | `/api/local-profit-data` | `/api/profit-analysis/payment-date`와 별개 |
| registerReload/registerRenderer 키 | `local_profit_data` (menuKey 기반) | 자동으로 겹치지 않음 |
| `listGridApi`/`loadLocalProfitData` | 파일 내부 IIFE 클로저 | 다른 파일과 이름이 같아도 무관 |

## 7. 검증 체크리스트

- [ ] 4절 DDL을 로컬DB에 실행했는지
- [ ] "손익현황(로컬)" 탭 최초 오픈 시 빈 그리드만 뜨고 자동조회는 안 되는지 (의도된 동작)
- [ ] "조회" 버튼 클릭 시 그리드에 데이터가 채워지는지
- [ ] 검색조건 필드에서 Enter 입력 시에도 동일하게 조회되는지
- [ ] 잘못된 날짜(예: `20260231`) 입력 시 "잘못된 날짜 형식입니다." alert만 뜨고 무한루프/중복 alert 없는지
- [ ] 탭을 닫을 때 콘솔 에러 없이 정리되는지 (`destroy()` 호출 확인)
- [ ] `logs/app.log`에 `손익현황(로컬) 조회 완료(date=...): N건` 로그가 남는지
- [ ] 브라우저 Network 탭에서 `content-renderers/local-profit-data.js`가 탭을 열기 전까진 요청 안 나가는지 (Lazy Loading 확인)

## 8. 알려진 제약 / 다음에 할 일

- 그리드에서 특정 행 선택 시 상세 화면으로 이동하는 기능 없음 (단일 그리드, 유형 A)
- `local_profit_data` 테이블에 실제 데이터를 채우는 배치/동기화 로직은 이 문서 범위 밖 (수동으로 INSERT 필요)
- 삭제/저장 기능 없음 (조회 전용 화면)
